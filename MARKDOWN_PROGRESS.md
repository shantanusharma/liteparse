>   1. **`infer_tracks_from_raw_items`**: walks forward from `start_idx` while `table_rows_adjacent` holds (cap `TABLE_TRACK_INFERENCE_MAX_ROWS=12`), collects `span.x` for every non-whitespace span on each row with ≥2 spans, then single-link clusters at `TABLE_TRACK_TOLERANCE_PT` (6pt). The Apple Q3 row alone exposes all 5 column anchors (49.5/119/152.6/186.2/253.3); even with PDFium-level merge on 9 Months row, the union across rows recovers the full geometry.
>   2. **Track-soundness guards** (all required): `tracks.len() ≥ TABLE_MIN_COLUMNS`, `tracks.len() > baseline_cells.len()` (only re-seed if we'd actually unlock more columns), and `min(inter-track gap) ≥ max(1.5 × seed_font_size, 12pt)` — rejects inferred "tracks" that are really inter-word positions in prose (word gaps are typically <1× font size).
>   3. **Strict seed-row check**: the seed (`lines[start_idx]`) must have `≥ tracks.len()` raw PDFium spans AND each span must single-cover (no track ranges where one span x-extent overlaps ≥2 anchors). This is the load-bearing false-positive guard — see "Seed strictness" below.
>   4. **`cells_from_raw_items_with_tracks`** per row: walk raw items, assign each to its nearest track within tol; for items whose `[x, x+width]` covers multiple tracks AND whose `x` anchors at the leftmost covered track (within tol), generalize `split_merged_at_anchor` to N anchors — pick whitespace positions whose linearly-interpolated x is closest to each subsequent track. Rejects rows with <2 PDFium spans (single-span lines spanning the track region are prose, not merged-numeric rows). Returns `Some(cells)` of length `tracks.len()` (some cells may be empty for sparse rows).
>   5. **Cell-text dedup via existing absorb path**: re-uses `absorb_header_lines` and the bold-first-row promotion already in `try_detect_table`. CV check via `row_spacing_cv` rejects runs with irregular cadence and falls back to the existing path.

> **Seed strictness was load-bearing.** The first cut required only `seed_spans + 1 ≥ tracks.len()` (one multi-cover split allowed in seed). Tightening to `seed_spans ≥ tracks.len()` AND no multi-cover spans in the seed eliminated the regression (multi_column held exactly at 49.4%, no `order` tests lost) while still catching Apple's Q3 row (5 spans, all single-cover at 5 tracks). > **2026-06-03 single-page chrome detector — olmOCR-bench headers_footers 38.9% → 58.6% (+19.7), overall 31.2% → 34.0% (+2.8).** The existing `compute_header_footer_set` is structurally inadequate on the bench's single-page inputs — cross-page repetition can't fire when there's only one page. Wired through `format_markdown` → `classify_page_with_filters` as a 6th param so the filter step strips both cross-page-matched lines AND per-page chrome before table detection / heading promotion runs.
>
> **Detection signal:** position band (top 15% OR bottom 15% of page) + isolation gap (≥ 1.0 × body_size to the nearest non-chrome-pattern neighbor in y-order) + chrome pattern signature. Patterns covered in `matches_chrome_pattern`: URL/DOI prefixes (`http://`, `https://`, `www.`, `doi:`, `doi.org/`, `dx.doi.org`), journal banner phrases (`please cite this article`, `contents lists available at`, `available online at`, `downloaded from`), copyright (`©`, `copyright `, `all rights reserved`), `Page N`/`Page N of M`/lone-digit page numbers (≤4 digits), `Vol.`/`Vol`/`Vol,` + digit (ASCII byte-scan, not regex), and journal-cite-style (4-digit year `19xx`/`20xx` + numeric `N-M` / `N–M` / `N—M` page range + ≤120 chars). 200-char ceiling on candidate strip so a body paragraph that happens to start with a URL isn't nuked. No regex dep — manual char/byte scans (one bug found and fixed during dev: byte-indexing into `&lower[i..]` panics on non-ASCII `–` in `Cell Chemical Biology 24, 1–9, …` — replaced with explicit ASCII-byte check).
>
> **Crucial isolation refinement:** the first attempt filtered isolation-neighbor candidates by "must be outside the position band," which made tightly-stacked top text always-isolated (no out-of-band neighbor exists). Real body chrome would then strip even when followed immediately by body content. Fix: filter by "must not match a chrome pattern itself," so multi-line chrome groups (e.g. `Please cite this article in press as: ... \n Cell Chemical Biology (2017), http://dx.doi.org/...`) are treated as one unit (both lines pattern-match), and isolation is measured against the first non-chrome body line. False-positive guard: required the pattern hint as a hard gate — initially also accepted a font-size delta from body, but dropped that because real headings/titles routinely satisfy it, and the bench has too many "preserve the title" cases to risk it.

> **2026-06-02 sparse-new-row path in borderless detector.** Targeted the "first data row gets eaten by the header" failure mode (doc 180, 045, 088/089/090, 127, 083, 084 …). Old logic in `try_detect_table` (`markdown_layout/tables.rs`) only had one branch for partial-cell lines — a wrap-merge that fired any time `centroid_dy ≤ 1.5 × line_height AND all_align_track`. That ate any data row whose cell count was less than the header's into the header itself, fragmenting tables with sparse columns.
> - **New branch** runs *before* the wrap path: when (a) all cells map to distinct column tracks, (b) `cells.len() >= 2`, and (c) `bottom_gap >= 0.5 × line_height` (clear inter-row spacing, not a stacked continuation), promote the partial line to a new row with empty `TableCell`s at the missing track positions.
> - **`cells.len() >= 2` is the load-bearing guard.** Without it, single-cell continuations like `"2022"` 5pt below `"1.0 April 30,"` get promoted to standalone rows (gap signal is noisy on a single cell). With it, 1-cell partial lines always fall through to the existing wrap-merge path (which is what we want — those are real continuations). The first attempt without this guard regressed TEDS by 0.04 across docs 130/051/082/090/147; adding it converted the net into +0.049.
> - **`bottom_gap >= 0.5 × line_height` discriminates "clear new row" from "wrap"** — a wrap typically overlaps the prior baseline (negative gap) or sits within ~0.3 × line_height of it. The 0.5 floor leaves a dead-zone (0.3, 0.5) that falls through to the existing wrap path, which is correct for the in-between cases.
> - **Wrap path unchanged.** First attempt tightened the wrap path's gap condition simultaneously, which regressed previously-clean tables (doc 130 0.84 → 0.10) because the existing wrap-merge was load-bearing for tightly-stacked multi-line cells. Reverted and reordered — sparse-new-row runs first, wrap path runs second with original conditions.

> **2026-06-01 column-start histogram fallback in XY-cut.** Targets the column-interleaving failure mode behind most remaining MHS=0 docs (decimal section heads on 2-column policy/academic/business reports — docs 029, 038, 039, 077, 189, …). The density-based V-cut search inside `xy_find_best_cut` fails on these layouts because a column-spanning element (wide heading, figure, table) fills the gutter at one y-band, inflating the density at the gutter's x and erasing the valley. Added `xy_find_column_cut` as a *fallback* consulted only when **both** `xy_find_best_cut` calls (H and V) return `None`.
>
> **Algorithm.** Histogram every item's left-edge x with 5pt buckets across the full item span (`min_x..max_x`, not bbox-relative — see the bbox aside below). Cluster contiguous occupied buckets (allowing ≤2-bucket gaps for intra-column drift) into peaks; each peak carries its summed item count. Then apply four conservative gates *in this order*:
> - `XY_COLUMN_MIN_LINES_PER_PEAK = 10` — drop peaks with fewer than 10 items. Indented paragraph-starts and footnote markers produce 3–6 item peaks; real column-left edges have 15+ items in a typical academic page.
> - **Exactly 2 strong peaks remaining.** 3+ peaks means tabular structure (one peak per cell column); we bail rather than slice a table.
> - `XY_COLUMN_PEAK_DOMINANCE = 0.55` — the two peaks together must hold ≥55% of all items. Rules out scattered-x prose layouts (lists, lab manuals, multi-indent docs) where the two tallest peaks are tall only relative to a flat tail.
> - `XY_COLUMN_PEAK_BALANCE_RATIO = 0.4` — smaller-peak / larger-peak count ratio ≥ 0.4. Filters out "single column with one off-x bullet stack" cases (90/10 split → false-positive column cut).
>
> Plus the existing geometric constraint: gap between the two peaks must be ≥ `XY_COLUMN_MIN_GAP_FRACTION = 0.25` × total item-span width. Cut position is the midpoint between the two peaks; returned `CutCandidate` carries `score = 1e9` so it beats every density-derived cut but still loses to `xy_find_banner_cut`'s `f32::INFINITY`.
>
> **Why histogram items, not lines.** The natural first cut was: y-band items into lines, take each line's `x_min`, histogram those. That fails on the *exact* layouts we're trying to fix — items from the left and right columns at the same baseline fold into one "line" whose `x_min` is the left-column edge and whose `x_max` reaches the right-column tail. The right column's left edge is hidden. Switching to per-item left edges lets each column's items contribute to their own peak directly.
